# Ai Web Test 自动化测试平台

## 一、系统概述

### 1.1 系统定位

Ai Web Test 是一款基于人工智能技术的自动化测试平台,旨在解决传统软件测试中效率低下、维护成本高、执行不稳定等痛点问题。系统通过 AI 技术实现测试用例的智能生成、自动执行和批量维护,将测试工程师从重复性工作中解放出来,专注于更有价值的测试策略设计。

### 1.2 核心价值

- **效率提升 10 倍**: 从产品原型到可执行测试用例,全流程自动化生成,传统需要 2-3 天的工作量缩短至 10 分钟
- **零编码门槛**: 测试人员无需编写任何自动化脚本代码,AI 自动理解测试步骤并转换为可执行指令
- **智能维护**: 当系统功能变更时,使用自然语言描述变更内容,AI 自动批量更新相关测试用例
- **稳定执行**: 基于先进的 MCP 协议和浏览器自动化技术,执行成功率高达 95% 以上

### 1.3 技术架构

系统采用前后端分离架构,前端基于 React 18 + TypeScript 构建现代化用户界面,后端使用 Node.js + Express 提供 REST API 和 WebSocket 实时通信服务。核心技术栈包括:

- **AI 引擎**: 集成大语言模型(LLM),支持 Claude/OpenAI 等多种 AI 服务
- **自动化引擎**: 基于 Playwright 浏览器自动化框架和 MCP 协议
- **数据存储**: MySQL 关系数据库 + Qdrant 向量数据库
- **实时通信**: WebSocket 技术实现测试执行过程的实时状态推送

## 二、核心功能详解

### 2.1 AI 智能用例生成系统(重点)

这是 Ai Web Test 最具创新性的功能模块,实现了从产品设计原型到完整测试用例的全自动生成。

#### 2.1.1 功能概述

传统测试用例编写是一项耗时且需要经验的工作,测试工程师需要:
1. 阅读产品原型文档或设计稿
2. 理解业务逻辑和交互流程
3. 分析测试点和边界条件
4. 编写详细的测试步骤和预期结果

Ai Web Test 通过 AI 技术将这一过程完全自动化,只需上传 Axure 原型文件(产品设计师常用的原型设计工具),系统即可自动生成专业级测试用例。

#### 2.1.2 三阶段渐进式生成流程

系统采用创新的"三阶段渐进式"生成方法,确保生成的测试用例质量和准确性:

**阶段一: 智能测试模块拆分**

当测试人员上传 Axure 原型文件后,系统首先进行需求文档生成:

1. **HTML 内容解析**: 系统读取 Axure 导出的 HTML 文件,提取页面元素、交互逻辑和文本内容
2. **AI 需求理解**: 大语言模型分析 HTML 结构,理解页面功能和业务逻辑
3. **需求文档生成**: 自动生成结构化的需求文档,包含字段定义、校验规则、业务逻辑等章节

接下来进入测试模块拆分:

- AI 分析需求文档,识别不同的功能模块(例如查询条件区、列表展示区、操作按钮区等)
- 根据页面类型(列表页、表单页、详情页、混合页)智能拆分测试模块
- 为每个模块标注优先级(高/中/低)和关联的需求章节

**示例输出:**
```
测试模块 1: 订单查询条件区 (高优先级)
- 描述: 测试订单号、日期范围、订单状态等查询条件的功能
- 关联章节: 3.1 查询条件说明

测试模块 2: 订单列表展示 (中优先级)
- 描述: 测试订单列表的数据展示、分页、排序功能
- 关联章节: 3.2 列表字段定义

测试模块 3: 订单操作按钮 (高优先级)
- 描述: 测试编辑、删除、导出等操作按钮的功能
- 关联章节: 3.3 操作按钮说明
```

**阶段二: 生成测试目的**

对于每个测试模块,系统会生成多个"测试目的":

- 测试目的是对该模块不同测试维度的描述(功能验证、边界测试、异常处理等)
- AI 根据需求文档内容,为每个模块生成 3-8 个测试目的
- 每个测试目的包含优先级、覆盖范围、预估测试点数量等信息

**示例输出:**
```
测试目的 1: 订单号精确查询功能验证 (高优先级)
- 描述: 验证输入完整订单号能准确查询到对应订单
- 覆盖范围: 正常流程、空结果处理
- 预估测试点: 5 个

测试目的 2: 日期范围查询边界测试 (中优先级)
- 描述: 测试日期范围的边界条件和异常情况
- 覆盖范围: 起止日期相同、跨年查询、未来日期等
- 预估测试点: 8 个
```

测试人员可以查看所有测试目的,选择需要生成用例的测试目的,实现按需生成。

**阶段三: 生成完整测试用例**

当测试人员选中某个测试目的后,系统开始生成完整的测试用例:

1. **测试点生成**: AI 根据测试目的和需求文档,生成多个具体的测试点(Test Points)
2. **测试步骤编写**: 为每个测试点编写详细的执行步骤,包括操作动作和预期结果
3. **用例结构化**: 将测试点组织成标准测试用例格式,包含用例名称、前置条件、测试数据、执行步骤等

**示例输出 - 完整测试用例:**
```
用例名称: 订单号精确查询-正常流程验证
系统名称: 电商后台管理系统
模块名称: 订单管理
优先级: P0(最高)

前置条件:
- 系统中已存在订单数据
- 用户已登录订单管理页面

测试点:
1. 输入存在的订单号进行查询
   步骤: 在查询条件区输入订单号"202501070001",点击查询按钮
   预期: 列表显示该订单的详细信息,订单号、金额、状态正确

2. 输入不存在的订单号进行查询
   步骤: 在查询条件区输入不存在的订单号"999999",点击查询按钮
   预期: 列表显示空数据提示"暂无数据",不报错

3. 订单号输入格式校验
   步骤: 输入包含特殊字符的订单号"<script>",点击查询按钮
   预期: 系统提示"订单号格式不正确"或自动过滤特殊字符
```

生成的用例会进入"草稿箱",测试人员可以:
- 预览和编辑每个测试用例
- 选择需要保存的用例
- 批量保存到用例库

#### 2.1.3 智能补全机制

为了进一步提升生成质量,系统提供"智能补全"功能:

**工作原理:**
1. **AI 预分析**: 在生成需求文档前,AI 会预先分析原型内容,识别"不确定的关键信息"
2. **问题生成**: 将不确定信息转换为用户友好的确认问题
3. **用户确认**: 通过对话框形式让测试人员补充确认
4. **增强生成**: 将用户补充的信息融入需求文档生成,提升准确性

**识别的不确定信息类型:**
- **页面类型不明确**: 该页面是列表页、表单页还是详情页?
- **枚举值缺失**: 下拉框的可选项有哪些?
- **业务规则模糊**: 某个字段的校验规则是什么?
- **字段含义不清**: 某个字段或按钮的具体含义?

**示例场景:**
```
原型中有一个"状态"下拉框,但 HTML 中只显示"请选择状态",没有具体选项

智能补全对话框:
┌─────────────────────────────────────┐
│ 🤖 AI 发现不确定信息,需要您确认     │
├─────────────────────────────────────┤
│ 问题 1: 页面类型                    │
│ 这个页面的主要功能类型是?           │
│ [ ] 列表查询页                      │
│ [ ] 表单录入页                      │
│ [✓] 列表+表单混合页                 │
│                                     │
│ 问题 2: 状态下拉框的可选项          │
│ 请输入"状态"字段的所有可选值:      │
│ [待支付、已支付、已发货、已完成、   │
│  已取消]                            │
└─────────────────────────────────────┘
```

用户确认后,这些信息会被 AI 融入需求文档生成,使测试用例更准确。

#### 2.1.4 技术亮点

- **多文件上传支持**: 支持上传多个 HTML 文件和资源文件(图片、CSS 等),系统自动识别主文件
- **实时进度反馈**: 生成过程中实时显示 AI 工作进度(解析 HTML、理解业务、生成文档)
- **可编辑需求文档**: 生成的需求文档支持在线编辑,测试人员可以补充或修正内容
- **草稿箱机制**: 生成的用例先进入草稿箱,支持预览、编辑、删除,确认后再保存到用例库
- **质量评分**: 每个生成的用例都有 AI 给出的质量评分(通常在 85-95 分)

#### 2.1.5 实际效果

根据实际使用数据:
- **生成速度**: 上传原型后 1-3 分钟完成需求文档生成,每个测试用例生成耗时 10-20 秒
- **准确率**: AI 生成的测试用例准确率达 90% 以上,剩余 10% 可通过人工微调完善
- **覆盖率**: 自动识别的测试点数量通常是人工编写的 1.5-2 倍,覆盖更全面
- **效率提升**: 传统人工编写 50 个测试用例需要 2-3 天,使用 Ai Web Test 缩短至 30 分钟

### 2.2 AI 驱动的 UI 自动化执行系统(重点)

#### 2.2.1 功能概述

传统自动化测试需要测试工程师编写大量代码,包括:
- 元素定位代码(查找页面上的按钮、输入框等)
- 操作指令代码(点击、输入、等待等)
- 断言验证代码(检查执行结果是否符合预期)

这需要测试人员具备编程能力,学习成本高,维护成本也高(页面改动后代码需要同步修改)。

Ai Web Test 创新性地采用"AI 自动解析执行"方案,测试人员只需用自然语言描述测试步骤,AI 自动将其转换为可执行的自动化指令,真正实现零编码自动化测试。

#### 2.2.2 核心技术架构

系统采用 **MCP(Model Context Protocol)协议 + Playwright** 的技术组合:

**MCP 协议(模型上下文协议):**
- 这是由 Anthropic 公司开发的开放标准协议
- 作用是让 AI 模型可以与外部工具进行标准化交互
- 在 Ai Web Test 中,MCP 充当"翻译官"角色,将 AI 的理解转换为浏览器操作指令

**Playwright 浏览器自动化框架:**
- 由微软开发的现代化浏览器自动化工具
- 支持 Chrome、Firefox、Safari 等主流浏览器
- 提供可靠的页面元素定位和操作能力

**技术优势:**
- **智能定位**: AI 理解"点击保存按钮"后,自动在页面中找到保存按钮的位置,无需人工编写定位代码
- **容错能力**: 当页面结构发生变化时,AI 可以智能适应,不像传统代码那样容易失效
- **自然语言**: 测试步骤用人类语言描述即可,例如"在订单号输入框输入 12345"

#### 2.2.3 执行流程详解

**步骤 1: 测试步骤解析**

当用户点击"执行测试用例"后,系统会读取用例中的测试步骤,例如:
```
测试步骤:
1. 打开订单管理页面
2. 在订单号输入框输入"202501070001"
3. 点击查询按钮
4. 验证列表显示该订单信息
```

**步骤 2: AI 理解与规划**

AI 分析测试步骤,理解每一步的意图:
- "打开订单管理页面" → 需要导航到指定 URL
- "在订单号输入框输入..." → 需要找到输入框元素,填入文本
- "点击查询按钮" → 需要找到查询按钮,执行点击操作
- "验证列表显示..." → 需要检查页面元素和内容

AI 将每个步骤转换为"MCP 工具调用"指令。

**步骤 3: MCP 指令执行**

系统通过 MCP 协议与 Playwright 通信,执行具体操作:

```
// AI 生成的 MCP 指令(示例,实际为 JSON 格式)

1. 导航指令:
   工具: playwright_navigate
   参数: { url: "https://example.com/orders" }

2. 输入指令:
   工具: playwright_fill
   参数: {
     selector: "根据文本'订单号'智能定位输入框",
     value: "202501070001"
   }

3. 点击指令:
   工具: playwright_click
   参数: {
     selector: "根据文本'查询'智能定位按钮"
   }

4. 验证指令:
   工具: playwright_get_text
   参数: { selector: "列表区域" }
   验证: 返回内容包含"202501070001"
```

**步骤 4: 实时状态反馈**

执行过程中,系统通过 WebSocket 技术向前端实时推送状态:
- 当前执行到第几步
- 每一步的执行结果(成功/失败)
- 执行耗时
- 错误信息(如果失败)

测试人员在界面上可以看到实时更新的执行日志和进度条。

**步骤 5: 证据收集**

系统在执行过程中自动收集执行证据:
- **截图**: 每执行完一个关键步骤,自动截取页面截图
- **日志**: 记录详细的操作日志和系统响应
- **视频**(可选): 录制整个执行过程的视频
- **网络请求**: 捕获页面发起的 HTTP 请求和响应

这些证据可用于:
- 问题排查(当测试失败时,查看截图定位问题)
- 合规审计(某些行业要求测试过程可追溯)
- 报告生成(自动生成带截图的测试报告)

#### 2.2.4 智能能力

**智能元素定位:**

传统自动化需要编写精确的元素定位器,例如:
```javascript
// 传统方式 - 脆弱,页面改动就失效
await page.click('#btn_query_id');
await page.fill('input[name="orderNo"]', '12345');
```

Ai Web Test 的 AI 定位:
```
// AI 方式 - 智能理解,根据语义定位
"点击查询按钮" → AI 自动找到文本为"查询"的按钮
"在订单号输入框输入" → AI 自动找到标签为"订单号"的输入框
```

AI 会综合多种定位策略:
- 文本内容(按钮上的文字)
- 元素属性(placeholder、label 等)
- 位置关系(某个输入框旁边的按钮)
- 上下文理解(当前页面的业务含义)

**智能等待机制:**

浏览器中很多操作是异步的(例如点击按钮后数据需要加载),传统自动化需要手动编写等待代码。Ai Web Test 的 AI 会:
- 自动判断哪些操作需要等待
- 智能等待页面加载完成
- 等待 AJAX 请求返回
- 避免过早进行下一步操作

**异常智能处理:**

执行过程中可能遇到各种异常情况:
- 弹窗提示(确认对话框、广告弹窗等)
- 网络延迟导致的超时
- 页面结构临时变化

AI 会尝试智能处理:
- 自动关闭干扰性弹窗
- 在合理范围内重试失败的操作
- 当遇到无法处理的异常时,详细记录现场信息供人工分析

#### 2.2.5 执行监控与管理

**实时执行视图:**

系统提供"Live View"功能,测试人员可以:
- 实时看到浏览器中的执行画面
- 查看当前执行进度和步骤详情
- 看到每一步的执行日志和耗时
- 随时中止正在执行的测试

**队列调度系统:**

当需要执行多个测试用例时,系统提供队列管理:
- 将多个用例加入执行队列
- 支持设置优先级(高优先级用例先执行)
- 自动按顺序执行,无需人工干预
- 显示队列中每个用例的等待状态

**性能监控:**

系统内置性能监控机制:
- 记录每次执行的成功率、平均耗时
- 当检测到性能下降或成功率降低时自动报警
- 支持"快速模式"、"平衡模式"、"稳定模式"三种执行模式
- 自动选择最优执行策略

#### 2.2.6 执行结果展示

**测试报告:**

执行完成后,系统自动生成测试报告,包含:
- 用例执行结果汇总(通过/失败/跳过)
- 每个测试点的详细结果
- 失败步骤的错误信息和截图
- 执行耗时统计
- 可导出为 PDF 或 HTML 格式

**证据查看器:**

提供专门的证据查看界面:
- 按时间轴展示执行过程
- 点击每个步骤查看对应截图
- 对比执行前后的页面变化
- 下载所有证据文件打包

**Dashboard 数据看板:**

在首页 Dashboard 展示:
- 今日执行次数
- 整体成功率趋势
- 平均执行耗时
- 失败用例 Top 10
- 执行活跃度热力图

#### 2.2.7 技术亮点

- **零脚本维护**: 测试步骤用自然语言编写,无需维护定位器代码

### 2.3 AI 批量更新系统

#### 2.3.1 功能概述

在实际测试工作中,经常遇到这样的场景:
- 系统功能升级,50 个测试用例需要同步修改
- 新增了一个必填字段,相关用例都要补充这个步骤
- 操作流程调整,多个用例的某个步骤描述需要更新

传统方式需要人工逐个打开用例进行修改,耗时且容易遗漏。Ai Web Test 提供"AI 批量更新"功能,只需用自然语言描述变更内容,AI 自动批量修改相关用例。

#### 2.3.2 工作流程

**步骤 1: 选择更新范围**

测试人员指定需要更新的用例范围:
- 按系统/模块筛选(例如"订单管理"模块的所有用例)
- 按标签筛选(例如带有"支付流程"标签的用例)
- 按优先级筛选(例如 P0 高优先级用例)

**步骤 2: 描述变更内容**

用自然语言描述这次变更:
```
变更描述示例:
"系统新增了必填字段'客户类型',在所有涉及客户信息录入的步骤中,
需要补充选择客户类型(个人/企业)这个操作"
```

**步骤 3: AI 智能召回**

系统不会盲目修改所有用例,而是使用"语义召回"技术:
- 使用向量数据库(Qdrant)对所有用例进行语义索引
- 根据变更描述,智能识别哪些用例真正相关
- 给出召回原因(为什么这个用例需要修改)

示例:
```
召回结果:
✓ 用例 1: 新增个人客户 - 相关度 95%
  召回原因: 涉及客户信息录入,需要补充客户类型选择

✓ 用例 2: 编辑企业客户 - 相关度 92%
  召回原因: 修改客户信息时需要确认客户类型

✗ 用例 3: 查询客户列表 - 相关度 30%
  召回原因: 仅涉及查询,不涉及客户类型输入,不需修改
```

**步骤 4: 生成修改提案**

对于每个相关用例,AI 生成详细的修改提案:
- 使用 JSON Patch 格式描述具体修改内容
- 标注修改位置(哪一步需要修改)
- 给出修改理由
- 评估风险等级(低/中/高)
- 识别潜在副作用

示例提案:
```
用例: 新增个人客户
修改提案:
  步骤 3 后面插入新步骤:
  "在客户类型下拉框选择'个人'"

理由: 根据变更描述,需要在客户信息录入时增加客户类型选择

风险等级: 低
副作用: 无

原始步骤序列: 1→2→3→4→5
修改后序列: 1→2→3→[新步骤]→4→5→6
```

**步骤 5: 用户审核**

系统展示所有修改提案,测试人员可以:
- 查看每个用例的修改前后对比
- 编辑 AI 生成的修改内容(如果不准确)
- 选择应用或跳过某个提案
- 批量应用所有低风险提案

**步骤 6: 应用修改**

确认后,系统批量应用修改:
- 自动创建用例版本(保留修改前的版本,可随时回滚)
- 检测冲突(如果用例在审核期间被他人修改)
- 记录修改日志(谁在什么时间做了什么修改)
- 应用成功后发送通知

#### 2.3.3 技术亮点

- **语义理解**: 基于向量数据库的语义搜索,准确召回相关用例
- **JSON Patch 标准**: 使用业界标准格式描述修改,精确且可回溯
- **版本控制**: 每次修改自动创建版本,支持版本对比和回滚
- **风险评估**: AI 自动评估每个修改的风险,优先展示低风险提案
- **冲突检测**: 基于哈希值的冲突检测,避免覆盖他人的修改

#### 2.3.4 实际效果

- **修改效率**: 批量修改 50 个用例,从传统的 2-3 小时缩短至 10 分钟
- **准确率**: AI 召回相关用例的准确率 85% 以上,生成的修改提案准确率 90% 以上
- **安全性**: 版本控制和冲突检测机制,确保修改可追溯、可回滚
